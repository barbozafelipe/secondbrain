### Dynatrace → Gringo (GCP) — Leonardo
> Objetivo: enviar logs e métricas do ambiente GCP da Gringo para o Dynatrace.

| Componente | Status |
|---|---|
| Logs dos pods | ✅ Aparecendo no Dynatrace e Datadog |
| Service D3 | ⚠️ Ainda não aparece — ajuste pendente |
| GCP Monitor (agente dentro do GKE Autopilot) | 🔴 Pod não sobe — provável restrição do Autopilot |

**Problema**: o script de instalação do GCP Monitor precisa rodar de dentro do cluster a partir de uma máquina Linux com acesso à rede da Gringo via VPN.

**Solução aprovada**: subir VM Linux temporária **na AWS Zapay**, usar a VPN da Gringo para acessar o cluster, rodar o script, matar a VM.

**Próximo passo**: se o problema for mesmo do Autopilot, o cluster GKE Standard de staging (que Wellington está montando) será o ambiente de teste ideal.

### Rancher ← Clusters da Zapay e Gringo — Wellington + Thiago
> Objetivo: centralizar RBAC e visibilidade dos clusters adquiridos dentro do Rancher da Sem Parar.

**Bloqueio**: o Rancher da Sem Parar é privado. O agente precisa bater na URL da API do Rancher, mas existe overlap de rede entre a VPC da Zapay e a rede interna.

**Caminhos sendo explorados**:

**Decisão**: Wellington vai levantar com o time de networking. Gringo tem endpoint público no GKE, mas também precisa de VPN para o agente chegar no Rancher.

### HPA — Autoscaling por HTTP Requests (Gringo GKE) — João Pedro + Thiago + Leonardo
> Problema crítico que está gerando salas de crise.

- Antes da migração, serviços (ex: Gater, debate-service) escalavam via **métrica de HTTP requests** configurada no New Relic
- Após migração para GKE: sem essa métrica → **escala só por CPU/memória → subescala em picos de tráfego**
- Resultado: picos de response time → taxa de abandono subindo na Gringo

**Solução planejada**: configurar HPA com **PromQL equivalente** usando Cloud Monitoring do Google (tem Prometheus nativo integrado). Não usar Dynatrace para evitar dependência de ferramenta.

**Workaround atual**: réplicas mínimas elevadas manualmente (Léo já fez ajuste no debate-service).

**Próximo passo**: Thiago, Léo e Felipe olharem isso, testarem em stage manualmente via `kubectl`, validar e replicar via Helm/templates nos demais serviços afetados.

## 📌 Glossário técnico desta reunião

| Termo | O que é |
|---|---|
| **GKE Autopilot** | Kubernetes gerenciado pelo Google sem acesso ao master — restringe configurações como nodeSelector/toleration |
| **GCP Monitor** | Agente Dynatrace que roda dentro do cluster para coletar métricas de serviços GCP (Pub/Sub etc.) |
| **HPA** | Horizontal Pod Autoscaler — escala pods automaticamente por métrica |
| **Harbor** | Registry de imagens Docker usado pela Sem Parar |
| **Argo / ArgoCD** | GitOps — sincroniza estado do cluster com o repositório Git |
| **ImageStream** | Conceito do OpenShift para versionamento de imagens — não existe no Kubernetes padrão |
