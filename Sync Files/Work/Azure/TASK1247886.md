## 🏆 Resumo da Execução e Aprendizados Técnicos (Atualização Final)

**Status Final:** ✅ Concluído com Sucesso nos 3 ambientes (NPROD, HML e PRD).
**Versão Utilizada:** PostgreSQL 18.

### 🏛️ Arquitetura de Rede Implementada (Subnet Delegation)
- O PostgreSQL Flexible Server exige uma **Subnet Delegada** exclusivamente para o serviço `Microsoft.DBforPostgreSQL/flexibleServers` com a permissão (action) de `join/action`.
- **HML e NPROD:** Como compartilham a mesma VNet base (`stp-dig-vnet-nprd`), nós apontamos o banco de HML para a **mesma** subnet delegada do NPROD (`stp-dig-snet-psql-nprd`).
- **PRD:** O provisionamento da subnet de PRD (`stp-dig-snet-psql-prd`) foi feito injetando o novo bloco diretamente na variável `subnets` do arquivo base da rede (`main.tf`), garantindo o vínculo (association) com a Route Table legada (`stp-dig-rt-prd`).

### 🌐 Zona de DNS Privado (FQDN)
- Para isolar o tráfego interno, cada banco ganhou sua própria *Private DNS Zone*.
- O sufixo oficial da Azure é `.postgres.database.azure.com`, então prefixamos com a nossa convenção para evitar colisões: `stp-dig-aiagentsapp-hml-nprd...`.
- **Best Practice de Segurança (Firewall):** Sempre passar o **FQDN** para o time de Segurança liberar no Firewall, **nunca IPs estáticos**. A Azure gerencia a subnet dinamicamente, e o IP de uma instância pode mudar em caso de *failover* ou manutenção da Microsoft.

### 🛠️ Troubleshooting Terraform Resolvidos
1. **Erro de Versão de Provider (`no available releases match the given constraints`):**
   - Causa: Conflito de exigências de versão em códigos legados (`~> 3.0` vs módulos pedindo `~> 4.0`).
   - Solução: Ajustar o arquivo `terraform.tf` / `versions.tf` para usar `>= 3.0`, flexibilizando o download.
2. **Caminhos de Módulos Quebrados (`Unreadable module directory`):**
   - Causa: Em código antigo de rede de PRD, o caminho `source = "../../MODULES/vnet"` estava apontando para o diretório errado.
   - Solução: Adicionado o nível extra faltante (`../../../MODULES`).
3. **State Blob Locked:**
   - Causa: Cancelamento abrupto de execução (`Ctrl+C` x2). O Terraform bloqueia a Storage Account para impedir corrupção do `.tfstate`.
   - Solução: Utilizar `terraform force-unlock <LOCK_ID>`.

### 🛡️ Autenticação via Microsoft Entra ID (SecOps)
- Para alinhar com as melhores práticas de segurança (Zero Trust), desativamos a autenticação por senha local (`administrator_password -> null` e `password_auth_enabled = false`).
- Habilitamos o acesso exclusivamente via identidades corporativas (`active_directory_auth_enabled = true`).
- Criamos o grupo **`BR_AIAGENTS_DBA_CONTRIBUTOR`** no Entra ID e o definimos como Administrador Oficial do PostgreSQL via Terraform (recurso `azurerm_postgresql_flexible_server_active_directory_administrator`).

### 🕵️ Troubleshooting Avançado: Bloqueio de Rede no Entra ID (Outcome: 30)
- **Problema:** Ao atrelar o grupo Entra ID como administrador no banco de PRD, a API da Azure retornou o erro `Outcome: 30 (AadAuthPrincipalCreationFailed)`.
- **Causa Raiz:** A Tabela de Rotas de PRD (`stp-dig-rt-prd`) possuía uma rota default (`0.0.0.0/0`) que enviava todo o tráfego de saída para um Firewall/Appliance (`10.17.128.4`). Esse firewall estava bloqueando a comunicação do Control Plane do PostgreSQL com o `graph.microsoft.com` (necessário para validar o Object ID do grupo).
- **Solução (UDR Bypass):** Criamos uma rota específica na tabela de rotas (`rt-azure-ad`) apontando o tráfego da Service Tag **`AzureActiveDirectory`** direto para o Next Hop **`Internet`** (Backbone seguro da Azure), resolvendo instantaneamente o bloqueio sem comprometer a segurança da VNet.
