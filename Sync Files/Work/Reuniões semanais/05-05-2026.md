## 🗂️ Projetos em andamento

### Rotação de CA — Afinz / Carvalt (Ítalo + Wellington + Essias)

> **Contexto**: o time está trocando a CA (Certificate Authority) interna. O impacto vai além da infra — atinge aplicações que fazem TLS contra serviços internos.

**Fluxo de impacto identificado por Ítalo**:

```
CA velha vence
  → Certificados assinados pela CA velha ainda são válidos enquanto a CA não vence
  → Quando a CA velha vencer, o emissor para de ser reconhecido
  → Aplicações Java (Kafka via TLS) que têm a CA hardcoded no JKS/keystore falham
```

**Dois cenários de cliente Kafka**:

| Cenário | Impacto da troca de CA |
|---|---|
| App dentro do cluster (comunicação interna via service account) | ✅ Não usa TLS de app para app — sem impacto |
| App fora do cluster conectando no endpoint Kafka via TLS | ⚠️ Precisa importar a nova CA no keystore (JKS) — senão falha quando a CA velha vencer |

**Sequência de ação definida**:
1. Time de infra troca o certificado do lado de cá primeiro — sem impacto imediato para os devs
2. Wellington aciona o Esaú para comunicar os times de dev afetados
3. Os devs importam a nova CA no keystore das aplicações que conectam no Kafka via TLS
4. Os devs podem fazer isso agora em paralelo — não precisa esperar a infra terminar

> [!tip] Ponto técnico importante (Ítalo)
> Se o dev importar a nova CA enquanto o certificado emitido pela CA velha ainda é válido, **não há quebra**. O certificado antigo continua sendo reconhecido. Só haverá problema se a CA velha vencer e o dev ainda não tiver importado a nova.

**Times que precisam ser avisados** (Wellington citou):
- Time do **Silvano** — usa Kafka intensivamente
- **Kleber, Amorim, Randel** — mencionados explicitamente

## 📌 Glossário técnico desta reunião

| Termo | O que é |
|---|---|
| **CA (Certificate Authority)** | Autoridade certificadora — assina os certificados usados em conexões TLS internas |
| **JKS (Java KeyStore)** | Formato de armazenamento de chaves e certificados usado por aplicações Java |
| **TLS** | Transport Layer Security — protocolo de criptografia e autenticação entre cliente e servidor |
| **mTLS** | Mutual TLS — autenticação dupla: cliente e servidor se autenticam mutuamente via certificado |
| **Kafka** | Sistema de mensageria distribuído — as apps se conectam ao broker via endpoint com TLS |
| **Service Account (K8s)** | Identidade para pods dentro do cluster — comunicação interna não usa TLS de aplicação |
| **Esaú** | Membro do time que tem contato direto com as frentes de desenvolvimento afetadas |
