## 🗂️ Projetos em andamento

### Gringo (GCP) — Recall de Spot Instances — João Pedro + Wellington

> Maior incidente da semana. Ocorreu segunda-feira.

**O que aconteceu**:
- Google fez um recall geral de instâncias Spot na região brasileira
- **26 worker nodes** do ambiente Gringo foram derrubados de uma vez
- Pods ficaram em `Pending` — node selector estava configurado como `required` para Spot, impossibilitando subida em instâncias provisionadas
- Time da Gringo começou a mexer nos workloads apavorado, sem entender a causa

**Solução aplicada** (Wellington + João Pedro):
```yaml
# Antes — pods só subiam em Spot (e ficavam em Pending se não houvesse)
nodeSelector:
  preemptible: "true"  # required

# Depois — preferência por Spot, mas sobe em qualquer nó se não houver
affinity:
  nodeAffinity:
    preferredDuringSchedulingIgnoredDuringExecution: ...
```

**Status**: ✅ Resolvido. João Pedro ajustou os manifests.

### CA Interna — Rotação e Data Crítica — Ítalo + Wellington + Essias

> [!danger] Data crítica: 28/06/2027 às 15h22
> Ítalo atualizou a Root CA hoje (05/05/2026). Todos os certificados internos emitidos a partir de agora expiram nessa data. **Começar a trocar o que for possível agora**, para não acumular na data de corte.

**O que Ítalo fez**:
- Root CA atualizada → novos certificados têm validade de mais 5 anos
- Levantamento de todos os certificados possivelmente ativos compartilhado com time de servidores e Guilherme

**Impacto crítico — Kafka + aplicações Java**:

| Cenário | Impacto |
|---|---|
| App dentro do cluster → comunica via **service** (não usa TLS de app para app) | ✅ Sem impacto |
| App fora do cluster → conecta no **ingress/endpoint do Kafka via TLS** | ⚠️ Precisa importar a nova CA no JKS/truststore |

**Lógica do impacto** (Ítalo):
- Trocar o certificado do front não causa problema imediato
- Quando a CA velha vencer: emissor para de ser reconhecido → apps falham na validação
- Se o dev importar a nova CA agora (enquanto certs antigos ainda são válidos) → **não quebra nada**

**Plano de ação**:
1. Infra troca certificados do lado de cá ← primeiro
2. Wellington aciona **Isaú** para comunicar as frentes de dev afetadas
3. Time do **Silvano** (usa Kafka intensamente) → prioridade para avisar
4. Devs importam nova CA no JKS/truststore → podem fazer em paralelo já

## 📌 Glossário técnico desta reunião

| Termo | O que é |
|---|---|
| **Spot Instance** | VM com desconto que pode ser interrompida pela cloud a qualquer momento — Google fez recall em massa |
| **Node Selector `required`** | Configuração K8s que exige que o pod suba em um tipo específico de nó — sem alternativa se não houver |
| **Node Selector `preferred`** | Dá preferência a um tipo de nó, mas permite subi em outro se necessário |
| **Account Manager (Google)** | Ponto de contato da fabricante para acompanhar a conta — Gringo estava sem um há 6 meses |
| **CA (Certificate Authority)** | Autoridade certificadora — emite e valida certificados TLS internos |
| **JKS / Truststore** | Formato Java de armazenamento de certificados — apps configuram a CA aqui para validar TLS |
| **Kafka** | Sistema de mensageria — apps Java conectam ao broker via TLS e precisam reconhecer a CA |
| **IAP (Identity-Aware Proxy)** | Serviço Google que permite acesso seguro a recursos sem VPN |
| **Cloud NAT** | Serviço GCP de NAT para saída de tráfego de VPCs sem IP público |
| **Isaú** | Membro do time SRE — ponto de contato para comunicar frentes de dev sobre a rotação de CA |
