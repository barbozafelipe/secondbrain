## 🧠 Entendendo a Task

> *Antes de executar, entenda. O cérebro retém melhor quando conecta o novo ao que já conhece.*

### Por que essa task existe?

O **Hotsite Santander** estava sofrendo um **ataque de força bruta**: requisições em massa para validar se um cliente é válido dentro dos nossos sistemas, via API de **Motor de Crédito**. Sem limite de uso, o atacante consegue testar muitos CPFs/dados em sequência e ainda sobrecarrega a API.

A mitigação é amarrar o consumo dessa API a um **Usage Plan** bem restritivo (rate/burst/quota baixos) atrelado a uma **API Key** — assim o tráfego legítimo passa controlado e o abuso é estrangulado pelo throttling do próprio API Gateway.

### O que foi configurado, exatamente?

Trabalho no **API Gateway** na conta **AWS STP PRD (867102406853)**. Fluxo: **Criar Plano → Associar API → Criar Chave**.

| Configuração | Valores |
|---|---|
| **Usage Plan** | `plan_MotorCredito_Santander` <br> Rate: **3** / Burst: **6** / Quota: **10000 por dia** |
| **Stage associado** | `MotorCredito` — Stage `v1` |
| **API Key** | `key_MOTORCREDITO_HotsiteSantander` (Create and add new key) |

> 💡 **Âncora mental:** o Usage Plan é o "contrato" (limites de throttling), a API Key é a "assinatura" (acesso). Rate/Burst baixíssimos (3/6) são propositais — é mitigação de incidente, não capacidade de produção normal.
